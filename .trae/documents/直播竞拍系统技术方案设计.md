# 直播竞拍全栈系统 - 技术方案设计与数据库设计

## 一、项目概述

### 1.1 项目目标
构建一个高并发、实时性强的直播竞拍系统，支持数百用户同时在线出价，实现秒级数据同步，营造紧张刺激的竞拍氛围。

### 1.2 核心特性
- 0元起拍，固定加价幅度
- 封顶价自动成交
- 结束前出价自动延时10-30秒
- 分布式锁保证出价幂等性
- WebSocket房间级隔离，支持1000+用户同时在线

## 三、核心业务流程设计

### 3.1 竞拍全流程
```
商品上架 → 配置竞拍规则 → 开始竞拍 → 用户出价 → 
实时排名更新 → 倒计时 → 自动延时 → 竞拍结束 → 生成订单
```

### 3.2 出价核心流程（高并发场景）
```
1. 用户发起出价请求
   ↓
2. 后端校验：竞拍是否进行中？用户是否已登录？出价是否合法？
   ↓
3. Redis分布式锁（Lua脚本原子操作）
   ↓
4. 校验当前价格 + 加价幅度 == 用户出价
   ↓
5. 更新Redis中的当前最高价、最高出价人
   ↓
6. 记录出价历史（异步写入MySQL）
   ↓
7. 判断是否触发封顶价 → 是则立即结束竞拍
   ↓
8. 判断是否在结束前N秒内出价 → 是则自动延时
   ↓
9. WebSocket广播最新价格、排名、倒计时
   ↓
10. 释放分布式锁
```

### 4.2 Redis 数据结构设计

| `auction:product:{productId}` | Hash | 商品实时状态（当前价、最高价ID等） |

## 五、核心技术难点解决方案

### 5.1 分布式锁 + Lua 脚本保证出价原子性
```lua
-- 原子出价校验脚本
local productKey = KEYS[1]
local bidderId = ARGV[1]
local bidPrice = tonumber(ARGV[2])
local minIncrement = tonumber(ARGV[3])

-- 获取当前价格
local currentPrice = tonumber(redis.call('HGET', productKey, 'current_price') or 0)
local expectedNextPrice = currentPrice + minIncrement

-- 校验出价合法性
if bidPrice ~= expectedNextPrice then
    return redis.error_reply("INVALID_PRICE")
end

-- 更新状态
redis.call('HSET', productKey, 'current_price', bidPrice)
redis.call('HSET', productKey, 'last_bidder', bidderId)
redis.call('HSET', productKey, 'last_bid_time', redis.call('TIME')[1])

return redis.status_reply("OK")
```

### 5.2 WebSocket 房间广播机制
- 每个直播间对应一个 Socket.io Room
- 消息类型：`price_update`、`rank_update`、`countdown_tick`、`auction_ended`
- 心跳保活：30秒心跳，60秒超时断开

### 5.3 倒计时精确同步方案
- 服务端维护权威时间源
- 每秒通过WebSocket广播剩余秒数
- 客户端本地做视觉动画，以服务端时间为准
