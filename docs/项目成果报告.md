## 4. 核心功能清单

1. **用户注册与登录**：独立的商家端和用户端注册功能，用户名唯一性校验，bcrypt密码加密，注册成功直接签发JWT
2. **商品全生命周期管理**：商家端创建/编辑/上架竞拍商品，灵活配置 0 元起拍、加价幅度、封顶价、自动延时等规则
3. **高并发出价引擎**：Redis 分布式锁 + Lua 原子脚本保证 100+ 用户同时出价无冲突，无超卖无跳价，已通过完整压力测试验证
4. **实时同步直播竞拍**：Socket.io 房间级隔离，价格更新、新出价、倒计时、竞拍结束全场景消息毫秒级推送
5. **自动延时机制**：竞拍结束前 N 秒内有人出价，系统自动延长竞拍时间，营造紧张刺激氛围
6. **订单流转系统**：竞拍结束自动生成订单，完整支持待支付 → 已支付 → 已发货 → 已完成状态流转
7. **双端独立JWT体系**：商家端和用户端完全独立的两套JWT认证策略，数据100%隔离

## 5. 端到端使用流程

1. 用户可以自由选择注册账号，系统自动校验用户名唯一性，密码bcrypt安全加密
2. 商家使用账号登录 PC 管理后台，进入商品管理页创建新的竞拍商品，配置好起拍价、加价幅度和自动延时规则
3. 商家在商品列表页点击「初始化竞拍」和「开始竞拍」，进入实时监控页面观察倒计时启动
4. 用户使用账号登录移动端 H5，在首页看到正在直播竞拍的商品列表，点击进入商品详情页
5. 用户点击「立即出价」按钮，系统通过 Lua 脚本原子更新价格，实时推送到所有房间内客户端
6. 如果结束前用户持续出价，系统自动触发延时机制，延长竞拍时长
7. 倒计时归零或价格达到封顶价，竞拍自动结束，系统生成对应订单，所有用户同步收到竞拍结束通知

### 本地启动步骤
1. 使用 Docker 一键启动 MySQL 和 Redis
   ```bash
   docker run -d --name tiktok-mysql -p 3306:3306 \
     -e MYSQL_ROOT_PASSWORD=123456 \
     -e MYSQL_DATABASE=tiktok_auction \
     mysql:8.0

   docker run -d --name tiktok-redis -p 6379:6379 redis:7.2
   ```

2. 启动后端服务
   ```bash
   cd backend
   npm install
   cp .env.example .env
   # 配置数据库连接
   npm run prisma:generate
   npm run prisma:migrate:dev --name init
   npx ts-node src/scripts/init-data.ts
   npm run start:dev
   ```

3. 启动商家端
   ```bash
   cd frontend/merchant-admin
   npm install
   npm run dev
   ```

4. 启动用户端 H5
   ```bash
   cd frontend/user-h5
   npm install
   npm run dev
   ```

## 12. 关键工程难点与解决方案

### 难点 1：高并发下的出价原子性问题
- **挑战**：100+ 用户同时点击出价，极易出现价格跳变、超卖等竞态条件
- **解决方案**：采用 Redis 分布式锁 + Lua 脚本，在服务端单线程内原子完成「读取当前价格 → 校验新价格合法性 → 更新价格」整个流程，100% 避免竞态问题，已通过完整压力测试验证

### 难点 2：全量客户端实时同步一致性
- **挑战**：网络延迟不同步，导致不同客户端看到的当前价格不一致
- **解决方案**：以服务端 Redis 中的状态为唯一可信源，所有价格变更事件通过 Socket.io 房间广播统一分发，客户端只负责展示，不进行任何状态计算

### 难点 3：自动延时机制实现
- **挑战**：结束前 N 秒内出价要动态延长倒计时，不能出现倒计时回退的视觉体验问题
- **解决方案**：内存 Map 管理所有活跃定时器，在检测到出价满足延时条件时，先清除旧定时器，再重新启动新的倒计时，全程实时通过 WebSocket 同步剩余时间

### 难点 4：双端用户体系隔离问题
- **挑战**：商家账号和普通用户账号是两个完全独立的体系，不能混淆
- **解决方案**：完全分离两张数据库表（Merchant表 + User表），两套独立JWT策略和守卫，严格的权限边界
