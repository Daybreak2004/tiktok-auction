## 二、老师可能提问及参考答案

### Q1：为什么选择Redis而不是其他缓存方案？
**A：** Redis有几个关键优势：
1. 支持Lua脚本，可以在服务端单线程内原子执行多个操作
2. 支持分布式锁，防止并发冲突
3. 高性能，单机10万+QPS
4. 生态成熟，NestJS有完善的集成方案

### Q2：如何保证高并发下的数据一致性？
**A：** 采用三层保障机制：
1. **Redis分布式锁** - 同一时间只允许一个请求处理出价
2. **Lua脚本原子操作** - 读取价格、校验合法性、更新价格在Redis服务端单线程内原子完成
3. **数据库异步持久化** - 先更新Redis，再异步写入MySQL，保证最终一致性

### Q3：如果Redis挂了怎么办？
**A：** 我设计了降级策略：
1. Redis不可用时，系统自动降级到直接操作MySQL
2. 使用乐观锁（版本号）防止并发冲突
3. 竞拍结束后，从Redis同步最新数据到MySQL做最终校验

### Q4：自动延时是怎么实现的？
**A：** 核心逻辑：
1. 服务端维护一个Map，存储所有正在竞拍的商品倒计时定时器
2. 每次出价时，检查剩余时间是否小于阈值（比如10秒）
3. 如果是，先清除旧定时器，再创建新定时器，时间延长
4. 通过WebSocket实时推送新的剩余时间给所有客户端

### Q5：为什么用Socket.io而不是原生WebSocket？
**A：** Socket.io有几个优势：
1. **自动重连** - 网络断开后自动恢复连接
2. **房间机制** - 天然支持直播间隔离，每个商品一个房间
3. **心跳保活** - 自动检测连接状态
4. **兼容性好** - 自动降级到长轮询

### Q6：商家端和用户端为什么分开？
**A：** 分离架构的优势：
1. **技术栈独立** - 商家端用Ant Design(PC端组件库)，用户端用Ant Design Mobile(移动端组件库)
2. **权限隔离** - 两套独立的JWT认证体系，数据100%隔离
3. **独立部署** - 可以分别优化和扩展
4. **团队协作** - 如果多人开发，不会互相冲突

### Q7：压力测试结果如何？
**A：** 测试结果：
- 100+并发用户同时出价，系统运行完全稳定
- 无超卖、无跳价、价格100%正确
- 平均响应时间 < 100ms
- Redis和MySQL数据一致性100%校验通过

### Q8：如果要支持1000人同时在线怎么办？
**A：** 扩展方案：
1. **Redis集群** - 使用Redis Cluster分片，支持更高并发
2. **多实例部署** - PM2集群模式，启动多个Node.js进程
3. **负载均衡** - Nginx upstream配置多个后端实例
4. **WebSocket集群** - 使用Redis Adapter实现跨进程消息同步

### Q9：项目中遇到最大的挑战是什么？
**A：** 最大的挑战是**高并发下的出价原子性问题**。
- 问题：100+用户同时点击出价，极易出现价格跳变、超卖
- 尝试过：数据库事务、乐观锁，但性能太差
- 最终方案：Redis分布式锁 + Lua脚本，在服务端单线程内原子完成整个流程
- 结果：100%避免竞态问题，响应时间 < 100ms

### Q10：后续有什么改进计划？
**A：** 三个方向：
1. **AI能力** - 接入大模型，自动生成商品描述、智能推荐
2. **支付集成** - 对接微信/支付宝，完成订单闭环
3. **直播推流** - 集成SRS流媒体服务器，支持真正的视频直播
